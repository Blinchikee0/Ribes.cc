Deploying a local model today demands DevOps skills — Python environments, CUDA versions, dependency roulette. Ribes reduces all of it to Install → Launch, taking on dependency resolution, process isolation, VRAM budgeting, and standardized status exchange.

— THE KEY THESIS OF THE RIBES ARCHITECTURE
Platform

The launcher does the DevOps.
You just run models.

Every model on Ribes ships with a signed manifest. The launcher reads it, checks your hardware, builds an isolated environment, and supervises the process — so nothing is left to a README.

One-click install

No Python, no CUDA hunting, no dependency hell. Ribes resolves an isolated environment per model with micromamba / uv, shares common base layers, and verifies every chunk by hash.

01

Hardware compatibility gate

Before you download a byte, the store checks the manifest against your real GPU, VRAM, RAM, and disk — and returns a plain verdict: fits, tight, or cpu-fallback.

02

Private by design

Inference happens on your hardware. Network egress is denied by default; every permission a model wants is declared in its manifest and approved by you — explicitly.

03

Live resource telemetry

VRAM, GPU utilization, tokens per second, and structured logs — streamed from every running model over a local bridge in real time, sourced from NVML and OS counters.

04

A real library

Delta updates, version rollback, file verification, and a VRAM-aware scheduler that never lets two models fight over the same memory — like a game launcher, for models.

05

UI even without a UI

Models that ship no frontend still get one: Ribes generates a clean chat or console interface from the input/output schema the model declares in its manifest.

Architecture

Four layers. One contract.

Chosen for minimal memory overhead — every megabyte the launcher eats is taken from the model — and for single-binary operations. The Rust core is the single source of truth for all state.

L1

Marketplace · cloud

Catalog, identity, payments, artifact hosting, versioning, and revenue share. Weights are served from S3-compatible storage behind a CDN with ranged, resumable HTTP — the API layer never becomes a traffic bottleneck.

GogRPC / WebSocketPostgreSQLRedis Pub/SubS3 + CDN
L2

Launcher · desktop client

A Tauri shell — native system webview instead of a bundled Chromium, so tens of megabytes of RAM, not hundreds. The long-running Rust + Tokio daemon downloads, verifies, provisions, supervises, and collects metrics without GC pauses.

TauriRust + TokioReact + TypeScriptSQLite
L3

Execution engine · sandbox

Process-level isolation with managed Python environments: direct GPU access with no container layer in the hot path, shared torch/CUDA base layers between models, and guaranteed kill-tree via Job Objects and cgroups.

micromamba / uvNVMLJob Objects / cgroupsVRAM scheduler
L4

SDK · developer standard

One declarative manifest plus an entrypoint that speaks the bridge protocol. It defines the contract that makes any AI project a predictable, installable artifact for the launcher.

Lifecycle

Seven states. One source of truth.

Every transition lives in the Rust core and is broadcast to the UI, the cloud, and the model process — so nothing ever drifts out of sync.

01

Resolving

The manifest arrives and the hardware gate runs: VRAM, RAM, disk, GPU vendor, and compute capability are checked against requirements before a single byte downloads.

HARDWARE GATE
02

Downloading

Chunked, resumable delivery from the CDN over parallel streams. Every chunk is verified against its SHA-256 before it is written into place; shared chunks are skipped via the dedup cache.

SHA-256 · RESUMABLE
03

Provisioning

An isolated environment is built from the manifest with micromamba / uv. Base layers are reused — the second model installs far faster than the first.

SHARED LAYERS
04

Registered

The model and its environment are recorded in the local SQLite database — the offline source of truth for your library — and appear ready to launch.

SQLITE
05

Launching

The scheduler confirms free resources, a sandboxed session is created with process limits, and the entrypoint is spawned. The IPC handshake opens the control channel.

SANDBOX · IPC
06

Running

The model reports ready, its UI becomes available, and metrics and logs stream live. System metrics are sampled independently by PID — even a silent model is monitored.

LIVE TELEMETRY
07

Stopping

Graceful shutdown with a deadline, then a forced kill of the whole process tree via Job Objects or cgroups. Every byte of VRAM is guaranteed to come back.

KILL-TREE

Local vs cloud

The honest trade.

Cloud inference still makes sense when you need scale beyond your own hardware. For everything else, the economics point at the GPU already under your desk.

CLOUD API

Renting inference

  • Billed per token or GPU-hour, forever
  • Prompts and files leave your machine
  • Queues and rate limits at peak hours
  • Scales beyond your own hardware
RIBES

Owning inference

  • Runs on hardware you already own — no per-token cost
  • Data never leaves the machine; egress denied by default
  • No queue — your GPU answers to you alone
  • Bounded by your VRAM — the hardware gate tells you before install
Ribes Bridge

A protocol, not a port scan.

The core and each model process talk over JSON-RPC 2.0 on a Unix domain socket or named pipe — no open TCP ports, invisible to the network, faster than localhost HTTP. Logs travel on a second channel: structured JSON lines with raw stdout/stderr captured as a fallback.

  • readymodel → coreUI type, UI address, declared I/O schema
  • statusmodel → coreloading · warming · idle · busy · error
  • progressmodel → coreweights loading into VRAM, 0–100
  • metricmodel → coretokens/s, iterations/s, queue depth
  • logmodel → corestructured level + message
  • request_inputmodel → coreask the user for data (generated UI)
  • outputmodel → coreinference result (generated UI)
  • shutdowncore → modelgraceful stop with a deadline
  • cancelcore → modelabort the current inference task

Security

      
Untrusted code, treated as untrusted.

      
A model is arbitrary code on your machine. Ribes assumes exactly that — and layers its defenses instead of hoping.

    

    

      

        

        
Egress denied by default
Outbound connections from a model are blocked unless declared in the manifest and approved by you.

      

      

        

        
Sandboxed processes
Each model sees only its working directory; the shared weight cache mounts read-only. Memory limits are enforced by Job Objects and cgroups.

      

      

        

        
Signed artifacts
Publishers are verified; artifacts are signed with the developer's key and every weight file carries a SHA-256 in the manifest.

      

      

        

        
Permission manifest
Network, filesystem paths outside the sandbox, multi-GPU — every capability is declared up front and granted explicitly.

      

      

        

        
Isolated third-party UI
Author-shipped web UIs render in a separate webview container with strict CSP, controlled egress, and no launcher API beyond granted permissions.

      

      

        

        
Guaranteed kill-switch
Stop sends a graceful shutdown with a deadline, then kills the entire process tree. VRAM is released — the system never hangs on a zombie.

FAQ

      
The honest engineering answers.

    

    

      

        Why Tauri instead of Electron?
        
Electron ships its own Chromium — 150–300+ MB of RAM per process. When the goal is to leave maximum VRAM and RAM to the model, that overhead is unacceptable. Tauri uses the system webview (WebView2 on Windows, WebKit on macOS/Linux), and all heavy logic — supervision, IPC, monitoring — lives in a Rust core with no GC pauses.

      

      

        Why not just use Docker containers?
        
On consumer Windows and macOS, full containers make GPU passthrough painful — WSL2, NVIDIA runtime forwarding — and add weight. The default is process-level isolation with managed Python environments built by micromamba / uv: direct GPU access, shared base layers, and a wheel cache. An optional container or micro-VM tier exists for untrusted or heavy cases.

      

      

        What data leaves my machine?
        
During inference — nothing. Prompts, images, and documents stay local. A model's outbound network access is denied by default and only opens if its manifest declares the permission and you approve it. Launcher telemetry is opt-in.

      

      

        What stops two models from crashing my GPU?
        
Consumer operating systems have no real VRAM isolation, so Ribes arbitrates: the VRAM-aware scheduler runs one GPU model at a time by default, keeps strict budget accounting from the hardware gate, and reserves memory for your system. If anything goes wrong, kill-tree via Job Objects or cgroups guarantees the memory comes back.

      

      

        Weights are tens of gigabytes. How is that manageable?
        
Content-addressed chunks with deduplication — weights shared between models download once. Updates ship as deltas, quantization variants (Q4 / Q8 / FP16) are separate downloads, and every transfer is ranged and resumable, verified chunk by chunk against SHA-256.

      

      

        What exactly does a developer have to ship?
        
A package with ribes.manifest.json, an entrypoint that connects the SDK bridge, per-platform dependency files (cuda / rocm / cpu), and the weights — which are chunked and hashed at publish. A web UI is optional: declare an I/O schema and Ribes generates the interface.
